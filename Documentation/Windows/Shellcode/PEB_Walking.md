# PEB Walking

## Purpose

PEB (Process Environment Block) walking is the foundational technique used in Windows shellcode to enumerate loaded modules and obtain their base addresses without relying on the Windows loader, imported function tables, or any fixed memory addresses. It is the bootstrap mechanism that makes all subsequent API resolution possible.

## Exploit Relevance

When shellcode executes, it arrives in memory as a raw byte sequence with no import directory, no relocations applied by a loader, and no guarantee about where in the process address space it resides. To call any Windows API, the shellcode must first find where kernel32.dll (or ntdll.dll) is mapped. PEB walking is the universally accepted solution to this problem and appears in virtually every non-trivial shellcode payload, including those generated by Metasploit, Cobalt Strike, and hand-written OSED-style exploits.

### The Alternative: Hardcoded Addresses

Before PEB walking became standard practice, some shellcode hardcoded the base address of kernel32.dll. On Windows XP (pre-SP2), kernel32 loaded at a predictable address (0x77E60000 on many builds). This approach fails for several reasons:

- ASLR (Address Space Layout Randomization), introduced broadly with Windows Vista, randomizes module base addresses on each boot.
- Even on pre-ASLR systems, different Windows builds, service packs, and hotfix levels change the load address.
- In a remote exploitation scenario the attacker cannot know the target's exact Windows build.
- Even if the attacker knows the build, High Entropy ASLR (Windows 8+) uses a much larger address space.

PEB walking solves all of these by reading the actual in-process module list at runtime, regardless of where modules are mapped.

## Technical Background: The Windows Process Structures

### Thread Environment Block (TEB)

Every thread in a Windows process has a Thread Environment Block (TEB), also called the Thread Information Block (TIB). The TEB is a per-thread structure maintained by the OS kernel and accessible at a well-known location via a CPU segment register:

- **x86**: The FS segment register is loaded by the kernel to point to the TEB of the currently executing thread.
- **x64**: The GS segment register plays this role.

The TEB layout (relevant fields for x86):

```
Offset  Size  Field
0x00    4     ExceptionList (pointer to SEH chain)
0x04    4     StackBase
0x08    4     StackLimit
0x0c    4     SubSystemTib
0x10    4     FiberData / Version
0x14    4     ArbitraryUserPointer
0x18    4     Self (pointer to TEB itself)
...
0x30    4     ProcessEnvironmentBlock (pointer to PEB)  <-- key field
```

The field at TEB+0x30 is a pointer to the Process Environment Block.

For x64, the TEB is the same conceptual structure but fields are wider (8 bytes for pointers) and the PEB pointer is at offset 0x60:

```
Offset  Size  Field
0x00    8     ExceptionList
0x08    8     StackBase
0x10    8     StackLimit
...
0x30    8     Self
...
0x60    8     ProcessEnvironmentBlock   <-- key field (x64)
```

### Process Environment Block (PEB)

The PEB is a per-process structure that contains information about the process: its image base, heap handles, environment variables, and crucially, the loader data that lists all loaded modules.

Relevant PEB fields (x86):

```
Offset  Size  Field
0x00    1     InheritedAddressSpace
0x01    1     ReadImageFileExecOptions
0x02    1     BeingDebugged
0x03    1     BitField (SpareBits, ImageUsesLargePages, etc.)
0x04    4     Mutant
0x08    4     ImageBaseAddress
0x0c    4     Ldr  (pointer to PEB_LDR_DATA)   <-- key field
0x10    4     ProcessParameters
...
```

The `Ldr` field at PEB+0x0C points to a `PEB_LDR_DATA` structure.

### PEB_LDR_DATA

```
Offset  Size  Field
0x00    4     Length
0x04    4     Initialized
0x08    4     SsHandle
0x0c    4     InLoadOrderModuleList.Flink       (LIST_ENTRY)
0x10    4     InLoadOrderModuleList.Blink
0x14    4     InMemoryOrderModuleList.Flink
0x18    4     InMemoryOrderModuleList.Blink
0x1c    4     InInitializationOrderModuleList.Flink  <-- classic walk
0x20    4     InInitializationOrderModuleList.Blink
```

### LIST_ENTRY and the Circular Doubly-Linked List

Windows uses `LIST_ENTRY` structures throughout the kernel and NT API to form doubly-linked circular lists:

```c
typedef struct _LIST_ENTRY {
    struct _LIST_ENTRY *Flink;   // Forward link (next element)
    struct _LIST_ENTRY *Blink;   // Back link (previous element)
} LIST_ENTRY;
```

The list is circular: the last element's Flink points back to the list head, and the head's Blink points to the last element. Walking the list stops when `Flink` equals the head address.

Each `LDR_DATA_TABLE_ENTRY` embeds three different `LIST_ENTRY` fields — one for each ordering (load order, memory order, initialization order). The pointer you get from following `Flink` in the `InInitializationOrderModuleList` is actually the address of the `InInitializationOrderLinks` field *within* an `LDR_DATA_TABLE_ENTRY`, not a pointer to the start of that entry. You must account for the offset when accessing other fields.

### LDR_DATA_TABLE_ENTRY (x86)

```
Offset  Size  Field
0x00    8     InLoadOrderLinks          (LIST_ENTRY)
0x08    8     InMemoryOrderLinks        (LIST_ENTRY)
0x10    8     InInitializationOrderLinks (LIST_ENTRY)  <-- list ptr lands here
0x18    4     DllBase                                  <-- +0x08 from InInitOrderLinks
0x1c    4     EntryPoint
0x20    4     SizeOfImage
0x24    8     FullDllName               (UNICODE_STRING)
0x2c    8     BaseDllName               (UNICODE_STRING) <-- +0x1c from InInitOrderLinks
```

`UNICODE_STRING` is:
```c
typedef struct _UNICODE_STRING {
    USHORT Length;          // byte count (not char count)
    USHORT MaximumLength;
    PWSTR  Buffer;          // pointer to wide-char string (no null terminator guaranteed)
} UNICODE_STRING;
```

For kernel32.dll: `BaseDllName.Buffer` = L"KERNEL32.DLL", `Length` = 24 (12 chars * 2 bytes each = 0x18).

## Step-by-Step PEB Walk

### Step 1 — Reach the TEB via FS

```asm
; x86: FS segment register points to the TEB
; TEB+0x30 = PEB pointer
mov eax, fs:[0x30]      ; dereference TEB+0x30 to get PEB address
```

The `fs:[0x30]` syntax is a memory reference using the FS segment as a base. The CPU computes the effective address as `FS_base + 0x30` and loads the DWORD there into EAX. No explicit base register is needed because FS is the implied segment.

For x64:
```asm
; x64: GS segment register points to the TEB
; TEB+0x60 = PEB pointer (8-byte pointer)
mov rax, gs:[0x60]      ; dereference TEB+0x60 to get PEB address
```

### Step 2 — Navigate to PEB.Ldr

```asm
; EAX = PEB base address
; PEB+0x0c = Ldr field (pointer to PEB_LDR_DATA)
mov eax, [eax + 0x0c]   ; EAX = PEB_LDR_DATA*
```

### Step 3 — Reach InInitializationOrderModuleList.Flink

```asm
; EAX = PEB_LDR_DATA*
; PEB_LDR_DATA+0x1c = InInitializationOrderModuleList.Flink
mov esi, [eax + 0x1c]   ; ESI = Flink of InInitializationOrderModuleList
                         ; ESI now points to InInitializationOrderLinks
                         ; within the first LDR_DATA_TABLE_ENTRY
```

**Why InInitializationOrderModuleList (the classic approach):**
Historically on Windows XP/Vista/7, the initialization order list contained:
- Index 0: ntdll.dll
- Index 1: kernel32.dll

This meant shellcode could skip to the second entry without comparing names. However this ordering is NOT guaranteed:
- Windows 8 added additional modules (kernelbase.dll) that changed the order.
- Windows 10/11 have more entries inserted early.
- The classic "skip 2" technique is unreliable on modern Windows.

The correct approach is to walk all entries and compare `BaseDllName`.

### Step 4 — Walk Entries and Compare BaseDllName

Each time we follow `Flink` from `InInitializationOrderModuleList`, we land on the `InInitializationOrderLinks` field within an `LDR_DATA_TABLE_ENTRY`. The relative offsets from `InInitializationOrderLinks` to other fields:

```
InInitializationOrderLinks is at LDR_DATA_TABLE_ENTRY + 0x10
DllBase        is at LDR_DATA_TABLE_ENTRY + 0x18  => InInitOrderLinks + 0x08
BaseDllName    is at LDR_DATA_TABLE_ENTRY + 0x2c  => InInitOrderLinks + 0x1c
```

So from the pointer ESI (which points to InInitializationOrderLinks):
- `[ESI + 0x08]` = DllBase
- `[ESI + 0x1c]` = BaseDllName.Length (USHORT)
- `[ESI + 0x20]` = BaseDllName.Buffer (PWSTR)

## Full Assembly Implementation

The following routine walks the PEB module list and returns the base address of kernel32.dll in EAX. Every line is commented.

```asm
; ============================================================
; find_kernel32 - locate kernel32.dll base via PEB walk
; Returns: EAX = kernel32.dll base address, 0 on failure
; Clobbers: EAX, EBX, ECX, ESI, EDI
; ============================================================

find_kernel32:
    xor     eax, eax                ; clear EAX (also avoids null bytes
                                    ; if using fs:[eax+0x30] form)
    mov     eax, fs:[eax + 0x30]    ; EAX = PEB (FS base + 0x30)
                                    ; using [eax+0x30] keeps encoding
                                    ; one byte shorter than fs:[0x30]
                                    ; and avoids the 0x30 null if needed
    mov     eax, [eax + 0x0c]       ; EAX = PEB.Ldr (PEB_LDR_DATA*)
    mov     esi, [eax + 0x1c]       ; ESI = InInitializationOrderModuleList.Flink
                                    ; ESI points to InInitOrderLinks
                                    ; within first LDR_DATA_TABLE_ENTRY

.next_module:
    ; --------------------------------------------------------
    ; ESI = current LDR_DATA_TABLE_ENTRY.InInitializationOrderLinks
    ; --------------------------------------------------------

    ; Get BaseDllName.Buffer
    ; BaseDllName is at InInitOrderLinks + 0x1c (the UNICODE_STRING struct)
    ; UNICODE_STRING.Length  is at BaseDllName + 0x00 (USHORT)
    ; UNICODE_STRING.Buffer  is at BaseDllName + 0x04 (PWSTR)
    mov     edi, [esi + 0x20]       ; EDI = BaseDllName.Buffer (wide string ptr)

    ; Get BaseDllName.Length
    movzx   ecx, word [esi + 0x1c]  ; ECX = BaseDllName.Length (bytes, not chars)

    ; Kernel32.dll = L"KERNEL32.DLL"
    ; Length = 12 chars * 2 bytes = 24 = 0x18
    cmp     ecx, 0x18               ; check expected length
    jne     .not_kernel32           ; wrong length, skip

    ; Compare name: "KERNEL32.DLL" in Unicode
    ; We compare 4 bytes at a time (2 wide chars per DWORD)
    ; L"KE" = 0x004B 0x0045 => little-endian DWORD = 0x0045004B
    ; L"RN" = 0x0052 0x004E => little-endian DWORD = 0x004E0052
    ; L"EL" = 0x0045 0x004C => little-endian DWORD = 0x004C0045
    ; L"32" = 0x0033 0x0032 => little-endian DWORD = 0x00320033
    ; L"." = 0x002E         => little-endian WORD  = 0x002E
    ; L"DL" = 0x0044 0x004C => little-endian DWORD = 0x004C0044
    ; L"L"  = 0x004C        => little-endian WORD  = 0x004C

    mov     eax, [edi + 0x00]       ; load first 4 bytes of name
    cmp     eax, 0x0045004B         ; compare "KE" (Unicode)
    jne     .not_kernel32

    mov     eax, [edi + 0x04]       ; next 4 bytes
    cmp     eax, 0x004E0052         ; compare "RN" (Unicode)
    jne     .not_kernel32

    mov     eax, [edi + 0x08]       ; next 4 bytes
    cmp     eax, 0x004C0045         ; compare "EL" (Unicode)
    jne     .not_kernel32

    mov     eax, [edi + 0x0c]       ; next 4 bytes
    cmp     eax, 0x00320033         ; compare "32" (Unicode)
    jne     .not_kernel32

    mov     eax, [edi + 0x10]       ; next 4 bytes: ".D"
    cmp     eax, 0x0044002E         ; compare ".D" (Unicode)
    jne     .not_kernel32

    mov     eax, [edi + 0x14]       ; final 4 bytes: "LL"
    cmp     eax, 0x004C004C         ; compare "LL" (Unicode)
    jne     .not_kernel32

    ; --------------------------------------------------------
    ; Found kernel32.dll
    ; DllBase is at InInitOrderLinks + 0x08
    ; --------------------------------------------------------
    mov     eax, [esi + 0x08]       ; EAX = DllBase (kernel32 base address)
    ret                             ; return with EAX = base address

.not_kernel32:
    ; --------------------------------------------------------
    ; Advance to next entry: follow Flink
    ; ESI = current InInitOrderLinks
    ; [ESI + 0x00] = InInitOrderLinks.Flink
    ; --------------------------------------------------------
    mov     esi, [esi]              ; ESI = next InInitOrderLinks (Flink)

    ; Check for end of list (Flink == list head address)
    ; The list head is PEB_LDR_DATA + 0x1c
    ; We do not have that address saved conveniently here,
    ; so we check for NULL (fragile) or use the InLoadOrder walk below.
    ; For the robust version, see the InLoadOrder walk.
    test    esi, esi                ; null check (defensive)
    jz      .not_found

    jmp     .next_module

.not_found:
    xor     eax, eax                ; return 0 (not found)
    ret
```

### Modern Robust Version: InLoadOrderModuleList Walk

The InLoadOrder walk is more reliable because it does not depend on initialization ordering. When using InLoadOrderModuleList, the `Flink` pointer lands on `InLoadOrderLinks` which is at offset 0x00 in `LDR_DATA_TABLE_ENTRY`, so offsets are different:

```
InLoadOrderLinks  is at LDR_DATA_TABLE_ENTRY + 0x00
DllBase           is at LDR_DATA_TABLE_ENTRY + 0x18  => InLoadOrderLinks + 0x18
BaseDllName       is at LDR_DATA_TABLE_ENTRY + 0x2c  => InLoadOrderLinks + 0x2c
  .Length                                                                  + 0x2c (USHORT)
  .MaximumLength                                                           + 0x2e (USHORT)
  .Buffer                                                                  + 0x30 (PWSTR)
```

```asm
; ============================================================
; find_kernel32_robust - InLoadOrderModuleList walk
; Saves list head to detect loop termination correctly.
; Returns: EAX = kernel32.dll base address, 0 on failure
; Clobbers: EAX, EBX, ECX, ESI, EDI
; ============================================================

find_kernel32_robust:
    ; --------------------------------------------------------
    ; Get PEB
    ; --------------------------------------------------------
    xor     eax, eax
    mov     eax, fs:[eax + 0x30]    ; EAX = PEB

    ; --------------------------------------------------------
    ; Get PEB_LDR_DATA
    ; --------------------------------------------------------
    mov     eax, [eax + 0x0c]       ; EAX = PEB.Ldr (PEB_LDR_DATA*)

    ; --------------------------------------------------------
    ; InLoadOrderModuleList head is at PEB_LDR_DATA + 0x0c
    ; Save head address in EBX for loop termination check
    ; --------------------------------------------------------
    lea     ebx, [eax + 0x0c]       ; EBX = &InLoadOrderModuleList (head)
    mov     esi, [eax + 0x0c]       ; ESI = InLoadOrderModuleList.Flink
                                    ; (first entry's InLoadOrderLinks)

.next_module_rob:
    ; --------------------------------------------------------
    ; Check if we have looped back to the list head
    ; --------------------------------------------------------
    cmp     esi, ebx                ; ESI == head? (end of list)
    je      .not_found_rob

    ; --------------------------------------------------------
    ; ESI = LDR_DATA_TABLE_ENTRY.InLoadOrderLinks (offset 0x00)
    ; BaseDllName.Length  at offset 0x2c
    ; BaseDllName.Buffer  at offset 0x30
    ; --------------------------------------------------------
    movzx   ecx, word [esi + 0x2c]  ; ECX = BaseDllName.Length

    cmp     ecx, 0x18               ; 24 bytes = "KERNEL32.DLL" (12 chars)
    jne     .advance_rob

    ; --------------------------------------------------------
    ; Case-insensitive compare: convert to uppercase on the fly
    ; Windows stores the name as "KERNEL32.DLL" (already uppercase)
    ; but we compare carefully to be safe.
    ; For a simple case: compare uppercase directly.
    ; --------------------------------------------------------
    mov     edi, [esi + 0x30]       ; EDI = BaseDllName.Buffer

    mov     eax, [edi + 0x00]
    or      eax, 0x00200020         ; force bits 5 of each char byte to 1
                                    ; converts ASCII uppercase to lowercase
                                    ; 'K'(0x4B)|0x20 = 'k'(0x6B)
    cmp     eax, 0x0065006B         ; "ke" lowercase
    jne     .advance_rob

    mov     eax, [edi + 0x04]
    or      eax, 0x00200020
    cmp     eax, 0x006E0072         ; "rn" lowercase
    jne     .advance_rob

    mov     eax, [edi + 0x08]
    or      eax, 0x00200020
    cmp     eax, 0x006C0065         ; "el" lowercase
    jne     .advance_rob

    mov     eax, [edi + 0x0c]
    ; digits: '3'(0x33) and '2'(0x32) — OR 0x20 does not affect digits
    cmp     eax, 0x00320033         ; "32"
    jne     .advance_rob

    mov     eax, [edi + 0x10]
    ; '.' = 0x2E, 'D'/'d' — handle mixed
    or      eax, 0x00200000         ; only uppercase second char (D)
    cmp     eax, 0x0064002E         ; ".d" (dot + lowercase d)
    jne     .advance_rob

    mov     eax, [edi + 0x14]
    or      eax, 0x00200020
    cmp     eax, 0x006C006C         ; "ll" lowercase
    jne     .advance_rob

    ; --------------------------------------------------------
    ; Match found — DllBase is at InLoadOrderLinks + 0x18
    ; --------------------------------------------------------
    mov     eax, [esi + 0x18]       ; EAX = DllBase
    ret

.advance_rob:
    mov     esi, [esi]              ; ESI = Flink (next entry)
    jmp     .next_module_rob

.not_found_rob:
    xor     eax, eax
    ret
```

## WinDbg Verification Workflows

### Verify PEB Address

```windbg
; Print the PEB address for the current process
!peb

; Alternatively, read TEB+0x30 manually
; (assuming 32-bit context)
dt ntdll!_TEB @$teb ProcessEnvironmentBlock

; Or read directly:
dd fs:[30] L1
```

### Walk PEB_LDR_DATA Manually

```windbg
; Get PEB address
r $t0 = @$peb

; Get Ldr pointer (PEB+0x0c)
r $t1 = poi(@$t0 + 0x0c)

; Print PEB_LDR_DATA
dt ntdll!_PEB_LDR_DATA @$t1

; Follow InInitializationOrderModuleList.Flink
r $t2 = poi(@$t1 + 0x1c)

; Print first LDR_DATA_TABLE_ENTRY (via InInitOrderLinks at offset 0x10)
dt ntdll!_LDR_DATA_TABLE_ENTRY (@$t2 - 0x10)
```

### Enumerate All Modules via lm

```windbg
; List all loaded modules with base addresses
lm

; Show kernel32 specifically
lm m kernel32

; Dump PEB loader list with module names
!peb
```

### Confirm Offsets with dt

```windbg
; Confirm TEB layout
dt ntdll!_TEB

; Confirm PEB layout
dt ntdll!_PEB

; Confirm PEB_LDR_DATA layout
dt ntdll!_PEB_LDR_DATA

; Confirm LDR_DATA_TABLE_ENTRY layout
dt ntdll!_LDR_DATA_TABLE_ENTRY
```

### Script: Walk the List and Print Module Names

```windbg
; Semi-automated walk using WinDbg script syntax
; First entry (InInitializationOrderModuleList)
r $t1 = poi(poi(@$peb + 0x0c) + 0x1c)
dt ntdll!_LDR_DATA_TABLE_ENTRY (@$t1 - 0x10)

; Second entry
r $t1 = poi(@$t1)
dt ntdll!_LDR_DATA_TABLE_ENTRY (@$t1 - 0x10)

; Continue following Flink...
```

## x86 vs x64 Differences

| Aspect | x86 | x64 |
|---|---|---|
| TEB segment register | FS | GS |
| TEB.PEB offset | 0x30 | 0x60 |
| Pointer size | 4 bytes | 8 bytes |
| PEB.Ldr offset | 0x0c | 0x18 |
| PEB_LDR_DATA InInitOrderModuleList.Flink offset | 0x1c | 0x30 |
| LDR_DATA_TABLE_ENTRY.DllBase (from InInitOrderLinks) | +0x08 | +0x10 |
| LDR_DATA_TABLE_ENTRY.BaseDllName.Buffer (from InInitOrderLinks) | +0x20 | +0x38 |
| UNICODE_STRING.Buffer offset within UNICODE_STRING | +0x04 | +0x08 |

### x64 Assembly (PEB Walk)

```asm
; x64 version — locate kernel32.dll
; Returns: RAX = base address

find_kernel32_x64:
    xor     rax, rax
    mov     rax, gs:[rax + 0x60]    ; RAX = PEB (GS:0x60)
    mov     rax, [rax + 0x18]       ; RAX = PEB.Ldr (x64 offset 0x18)
    mov     rsi, [rax + 0x30]       ; RSI = InInitializationOrderModuleList.Flink
                                    ; (x64: offset 0x30 in PEB_LDR_DATA)
    lea     rbx, [rax + 0x30]       ; RBX = list head (for termination)

.next_mod64:
    cmp     rsi, rbx
    je      .not_found64

    movzx   ecx, word [rsi + 0x28]  ; BaseDllName.Length (x64 offset from InInitLinks)
    cmp     ecx, 0x18
    jne     .adv64

    mov     rdi, [rsi + 0x30]       ; BaseDllName.Buffer (x64)

    mov     eax, [rdi + 0x00]
    cmp     eax, 0x0045004B         ; "KE"
    jne     .adv64

    mov     eax, [rdi + 0x04]
    cmp     eax, 0x004E0052         ; "RN"
    jne     .adv64

    ; ... continue comparison (same as x86 version) ...

    mov     rax, [rsi + 0x10]       ; DllBase (x64: InInitLinks + 0x10)
    ret

.adv64:
    mov     rsi, [rsi]              ; follow Flink (8 bytes in x64)
    jmp     .next_mod64

.not_found64:
    xor     rax, rax
    ret
```

Note that x64 does not have a pushad/popad instruction. Caller-save registers (rax, rcx, rdx, r8, r9, r10, r11) must be preserved manually if needed.

## Common Mistakes

### Mistake 1: Using Fixed Ordinal Position (Skip-N Logic)

Many tutorials show code that skips the first one or two entries without comparing names:
```asm
; WRONG — do not do this
mov esi, [eax + 0x1c]   ; InInitOrderModuleList.Flink
mov esi, [esi]           ; skip first (ntdll) 
mov ebx, [esi + 0x08]   ; assume this is kernel32 — WRONG on modern Windows
```
This relies on kernel32 being at a fixed position in the initialization order, which has not been reliable since Windows 8. Always compare `BaseDllName`.

### Mistake 2: Wrong Offset When Using InInitializationOrderLinks

A common off-by-one bug is forgetting that following `InInitializationOrderModuleList.Flink` lands on the `InInitializationOrderLinks` field at offset `+0x10` within `LDR_DATA_TABLE_ENTRY`, not the start of the structure. Applying offsets as if it were the start of the structure produces garbage reads:
```asm
; WRONG
mov esi, [eax + 0x1c]   ; Flink of InInitOrderModuleList
mov ebx, [esi + 0x18]   ; WRONG: treats InInitOrderLinks as entry start
                         ; DllBase from InInitOrderLinks offset is +0x08, not +0x18
```
Correct: `[esi + 0x08]` for DllBase when using InInitializationOrderLinks pointer.

### Mistake 3: Treating BaseDllName.Length as Character Count

`UNICODE_STRING.Length` is the length in **bytes**, not characters. L"KERNEL32.DLL" has 12 characters but `Length` = 24 (0x18). Code that compares `ecx, 12` instead of `ecx, 0x18` will never match and the walk will fall through:
```asm
; WRONG
cmp ecx, 12         ; length is in bytes, not chars — should be 0x18
jne .not_kernel32
```

### Mistake 4: Not Handling the Circular List Termination

The linked list is circular — if kernel32 is not found, the walk continues indefinitely or crashes when following a null/invalid pointer. Always check the termination condition by comparing `Flink` to the list head address, or at minimum checking for null:
```asm
; Minimal null check (not perfect but better than nothing)
test esi, esi
jz .not_found
```
The proper check compares `Flink` to the address of the list head in `PEB_LDR_DATA`.

### Mistake 5: Assuming UNICODE_STRING.Buffer is Null-Terminated

The `UNICODE_STRING` structure's `Buffer` field is not guaranteed to be null-terminated (the `Length` field defines the extent). Code that walks the string looking for a null terminator can over-read into adjacent memory. Always use the `Length` field to bound the comparison.

## Why the Classic Algorithm Can Fail on Modern Windows

**Windows 8 / Server 2012:**
KernelBase.dll was introduced and kernel32.dll now forwards many of its exports to it. The initialization order changed: ntdll, KernelBase, and kernel32 are all loaded early but not necessarily in any fixed sequence relative to each other.

**Windows 8.1+:**
Additional API set DLLs (api-ms-win-core-*.dll and ext-ms-win-*.dll) are virtual modules that appear in the loader lists. This increases the number of entries and changes positional assumptions.

**Windows 10 / 11:**
The loader performs parallel DLL initialization in some scenarios. The "initialization order" list reflects the actual order initialization completed, which depends on thread scheduling and is not deterministic across runs.

**The Bottom Line:**
The only reliable approach on all Windows versions from XP through 11 is to walk the entire module list (InLoadOrderModuleList is most straightforward) and compare `BaseDllName` explicitly against the target name. The comparison should be case-insensitive to handle any loader quirks.
